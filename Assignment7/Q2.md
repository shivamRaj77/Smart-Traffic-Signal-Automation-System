**Q2. Describe the following for your software engineering project.** 

### A) Implementation of Business Rules
Business rules in the Smart Traffic Signal Automation System dictate the core logic of the application. They have been implemented seamlessly across different architectural layers, primarily concentrating logic into specific module domains:

1. **Access Control & Authorization (Auth/Admin rules):**
   The system implements strict Role-Based Access Control (RBAC). Business rules regarding who can access and alter the backend systems are enforced within the `auth_service.py` and dedicated admin routes (`admin.py`). For instance, a regular `user` can view dashboards and simulations, whereas only an `admin` can apply traffic signal overrides (modifying light timings or signal modes).
2. **Traffic Signal Calculation Engine:**
   The ultimate business value of the application—dynamically allocating green-signal times based on congestion—is modularized in the `traffic_service.py`. The rule dictates that green time must be assigned proportionally to the calculated congestion factor, augmented by a baseline prediction model. These calculations are strictly capped by constants like `MIN_GREEN_TIME_SECONDS` and `BASE_SIGNAL_CYCLE_SECONDS` mapped out in the `config.py` module to prevent invalid physical signal states (e.g., setting a green light duration to 0 seconds or exceeding a maximum junction cycle).

### B) Implementation of Validation Logic
Yes, rigorous validation logic acts as the first line of defense before any processing occurs. The application leverages **FastAPI + Pydantic** to enforce robust validation on incoming data payloads.

* **Schema-Level Validation:** Before a request hits the application logic, Pydantic schemas mathematically inspect the structure. For example, in `schemas/auth.py`, the `RegisterRequest` forces strict validation limits:
  * Passwords must meet specific security criteria (`min_length=6`, `max_length=128`).
  * Usernames must be correctly sized (`min_length=3`).
  * User roles are evaluated through Regular Expressions (`pattern="^(user|admin)$"`), entirely rejecting unauthorized role types from being written to the database.
* **Domain Validation:** At the traffic analysis layer (`schemas/simulation.py`), inputs like `DirectionData` strictly dictate that `vehicle_count` must be castable to integers and `avg_speed` to floats, blocking corrupted or badly formatted types before the Traffic Service tries to feed them into the Machine Learning model (`predict_congestion`).

### C) Data Transformation implementation
Data transformation logic natively maps data sitting in the persistent storage layer to a clean, presentable JSON format consumed by the React/Vite frontend. The architecture handles transformation via multiple steps:

1. **Database Object mapping via DTOs (Data Transfer Objects):** 
   Internal database structures (`models/user.py`, `models/override.py`) contain sensitive or raw data such as hashed passwords or foreign key mappings. The system uses Pydantic Response schemas to transform and strip sensitive logic. 
2. **Schema Response Serialization:** 
   When sending a user profile, data instances are serialized exclusively through models like `UserResponse` (`schemas/auth.py`), mathematically dropping the `hashed_password` field and transforming the record into a frontend-safe dictionary. 
3. **Structured Composite Wrapping:** 
   In `traffic_service.py`, raw arrays, numerical outputs generated by the regression model (`Ridge`), and override overrides are algorithmically transformed and zipped into strongly typed JSON structures (`SimulationStepResponse`, `DirectionData`). The frontend purely receives an easily parsable object matching its UI component states for Dashboard charts, rather than having to decouple raw ML model outputs.
